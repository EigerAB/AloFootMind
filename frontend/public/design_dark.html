1.2 三层 RAG 语料策略

Layer 1
match_summaries
一场一条LLM · DeepSeek
跨事件整体理解，需要跨越数百条事件提炼语义，选用 LLM 生成自然语言摘要赋予语义检索能力。
Layer 2
tactical_segments
possession 切片规则模板
按 possession 切片，提取事件类型序列（Shot / Dribble / Carry / Under Pressure）、起止时间、场地分区（x: 0-40-80-120）。内容高度规律，避免 LLM 幻觉。
Layer 3
player_profiles
一人一条 · 赛季聚合LLM · DeepSeek
全赛季导入完成后 ingest_player_profiles() 跨场聚合 aggregate_player_season_stats，保证 profile 包含赛季全量视角。

1.3 幂等设计：ingestion_log 四步状态机

每场比赛在 ingestion_log 维护四个独立状态位，支持任意步骤断点续跑。

step_parsedonefailedpending解析原始 JSON → match_record
step_embeddonefailedpending← 写 Milvus 三层向量
step_postgresdonefailedpending← 写关系库
step_redis_invalidatedonefailedpending← 清 Redis key
每步前检查 is_step_done()；成功 → mark_step("done")；失败 → mark_step("failed",err) 并 return 跳过后续。重跑自动从第一个非 done 步续行。完成后按 rag:*:match:{id}:* 清 Redis。

向量检索层（RAG Service）

2.1 嵌入模型：BAAI/bge-m3 双向量

Dense Vector
1024 维
语义相似度 · COSINE 距离
捕捉上下文语义，适合开放式查询、模糊意图匹配
Sparse Vector
BM25-like
词频精确匹配 · IP 距离
适合专有名词：球员姓名、战术术语、赛事缩写
选用 bge-m3：单模型原生支持两种向量，无需维护两套模型；中英文跨语言检索，天然适配双语场景。

3.2 QA Sub-graph — 节点流（路由 + 检索图）

query_rewrite
实体识别 · 中→英改写
→
relevance_gate
双计数器路由决策
→
query_classify
多维意图分类
→
rag_retrieval
按 level 检索合并
→
END
↓ direct_answer → END（闲聊 ≤3轮）
↓ boundary_answer → END（边界提示）
query_rewrite
启动时从 PostgreSQL 加载实体字典 _ENTITY_DICT（内存缓存），最长子串匹配提取足球实体。命中实体或 query 含中文 → DeepSeek（temp=0.1, max_tokens=100）映射标准英文检索词，输出 rewritten_query。LLM 失败优雅降级为原始 query。
relevance_gate
双计数器：football_intent_count + generic_turn_count。有足球实体/关键词 → classify（重置 generic 计数）；无实体且 generic ≤ 3 → direct_answer（允许轻量闲聊）；无实体且 generic > 3 → boundary_answer（边界提示）。
rag_retrieval
按 query_levels 调用 retrieve()，使用改写后 query 和实体 ID 过滤，返回 rag_context。多 level 独立检索后合并按 score 降序。
Endpoint 流式生成
graph 返回后 endpoint 读取 _route 分三路调用异步 generator。System Prompt 三层分离：用户画像注入顶部 + RAG 上下文注入 Human 消息 + 最近 20 轮历史。解决"RAG 上下文污染对话记忆"。每轮结束后 messages + qa_meta 回写 PostgreSQL JSONB，下轮恢复意图计数状态。

POST /api/post-match→ task_id → SSE

用户选择一场已导入比赛，后台异步任务，前端 SSE 实时消费 Agent 执行日志。

Sub-graph 链路

fetch_match_data
SQL: matches JOIN teams JOIN events_aggregated
获取比赛元数据 + key_events_json
2
rag_retrieval（当场比赛维度）
tactical
retrieve(match_id=current_id, top_k=8)
player

3

DeepSeek LLM，注入比赛统计 + RAG 上下文（最多 20 条），输出结构化 Markdown 报告（比赛概述 / 战术阵型 / 关键时刻 / 球员亮点 / 战术总结）

4.2

赛前情报（pre_match）

POST /api/pre-match三维度 RAG

用户选择主客队，RAG 检索维度比 post_match 更广——涵盖 H2H 历史、主队近况、客队近况。

Sub-graph 链路

1
fetch_team_history
SQL: 查询两队最近 10 场交锋（h2h_matches），含阵型、比分、射门数
2
rag_retrieval（三维度）
match
retrieve(match_ids=[h2h_latest], top_k=2)，仅取最近一场 H2H 摘要
tactical
retrieve([h2h_latest, home_last, away_last], top_k=8)，H2H + 主客队近况
player
retrieve(player_ids=dedup(h2h+home+away), top_k=10)，三场关键球员合并去重
3
matchup_analysis
DeepSeek LLM，基于 H2H 历史 + RAG 上下文生成赛前情报报告（主队优势 / 客队优势 / 关键对决 / 战术预测）
降级策略：若两队无历史交锋，match_level 检索跳过，tactical_level 降级为仅主客队各自最近一场。

⚙️ 技术选型总结
模块技术选型理由
向量数据库Milvus原生 Hybrid Search，支持 Dense+Sparse 双向量，Dockerized 部署
嵌入模型BAAI/bge-m3单模型双向量，中英跨语言，无需额外 Sparse 引擎
Agent 框架LangGraph状态机驱动，Sub-graph 隔离，方便独立测试各分析路径
流式传输Fetch API + SSEPOST 请求传参（EventSource 仅支持 GET），token 级流式输出
幂等控制ingestion_log 四步状态位轻量 Saga 模式，无需消息队列，断点续跑，可清晰阐述
LLM 调用DeepSeek API批量 ETL 和实时对话统一使用
关系存储PostgreSQL赛事元数据、会话历史、分析报告，JSONB 存储 messages/qa_meta
缓存RedisRAG 结果缓存（TTL 600s）+ Agent 执行日志列表 + 任务状态
